## ⚙️ Windhawk 的技术核心分析

### 1. **即时 C++ 编译器：Clang / MSVC / TinyCC**

Windhawk 的“动态插件编译”本质上是：

* 在后台调用一个轻量级的 C++ 编译器（默认是 **Clang** 或 **MSVC Build Tools**）；
* 将用户的 `.cpp` 插件源文件编译成一个 `.dll`；
* 然后在运行时 `LoadLibrary()` 加载执行。

它做了很多优化来“假装”是即时的：

#### 🧠 编译优化策略：

* **最小编译单元**：每个插件是独立 `.cpp` 文件（无需复杂工程文件）。
* **内置头文件缓存**：Windhawk 自带了 `windhawk_api.h`（包含所有插件接口定义），编译器无需访问系统 SDK。
* **预编译头（PCH）或 Clang 模块缓存**：这样每次编译不需要重新解析 STL 或 Windows 头。
* **多线程并行编译**：在后台同时编译多个插件。

编译速度非常快（通常 < 1 秒）。

---

### 2. **安全的临时沙箱构建目录**

Windhawk 会将插件编译到一个临时目录，例如：

```
C:\Users\<User>\AppData\Local\Windhawk\Build\<PluginName>\
```

流程：

1. 用户保存 `.cpp`；
2. Windhawk 生成 `PluginName.dll`；
3. 写入编译日志；
4. 如果编译成功，立即卸载旧 DLL 并加载新 DLL；
5. 若编译失败，保留旧版本运行。

👉 这就是为什么 Windhawk 可以做到“**保存即生效**”。

---

### 3. **热重载机制**

编译后的 DLL 并不是直接替换内存中的代码，而是：

1. `FreeLibrary()` 卸载旧版本；
2. `LoadLibraryEx()` 加载新 DLL；
3. 调用导出的 `WindhawkPluginInit()` 或类似函数重新注册。

这种模式确保：

* 插件可以即时更新；
* 不需要重启主程序；
* 出错时能回滚。

---

### 4. **插件 API 的隔离与安全**

虽然 Windhawk 运行插件的代码是真正的本地 C++，但它通过以下方式降低风险：

| 机制            | 作用                                                     |
| ------------- | ------------------------------------------------------ |
| 🧩 API 封装     | 插件只通过 Windhawk 提供的 API（hook, log, config）访问系统，不直接触碰主进程 |
| 🧱 分层 Hook 架构 | 插件注入到目标进程，而非 Windhawk 主进程                              |
| 🧰 独立加载器进程    | 插件编译与运行在不同进程中，防止崩溃影响主程序                                |
| 🧩 只开放有限头文件   | 插件无法直接访问 Windhawk 内部类或函数                               |

简而言之：

> **Windhawk 允许“动态编译”，但通过隔离机制确保安全性和稳定性。**

---

### 5. **实际技术栈（核心组件）**

Windhawk 的构建和运行时组件大致包含：

| 模块    | 技术                                          |
| ----- | ------------------------------------------- |
| 编译器   | `clang-cl.exe` 或 `cl.exe`（MSVC Build Tools） |
| 构建系统  | 内置 CMake-like minimal builder               |
| 插件头文件 | `windhawk_api.h` 提供 `WINDHAWK_PLUGIN_*` 宏   |
| 运行时加载 | `LoadLibraryEx` + `GetProcAddress`          |
| 配置系统  | JSON/YAML 格式                                |
| 注入框架  | 类似 Microsoft Detours，用于 API Hook            |

Windhawk 插件通常长这样：

```cpp
#include <windhawk_api.h>

WINDHAWK_PLUGIN(L"记事本时间戳", L"在标题栏显示当前时间");
WINDHAWK_PLUGIN_VERSION(1, 0);

void Wh_OnPluginLoad() {
    // 插件逻辑
}
```

用户点“保存” → Windhawk 背后调用 Clang → 编译出 DLL → 自动加载。

> ⚡ **SymEngine：运行时用 LLVM JIT 直接在内存中生成机器码（JIT 编译）**

## 🧠 SymEngine 的 LLVM 动态编译机制概览

SymEngine 是一个 C++ 符号计算引擎（类似 Python 的 SymPy），但为了加速数学表达式执行，它允许把表达式“即时编译”为机器码。这是通过 LLVM 的 JIT 编译器完成的。

### 它的工作流程大致如下：

1. **生成 LLVM IR（中间表示）**
   SymEngine 在运行时把用户的数学表达式或函数翻译成 LLVM IR（不是 C++ 源码）。

2. **使用 LLVM JIT 编译器（ORC 或 MCJIT）**
   它调用 LLVM 的 JIT 引擎，将 IR 编译为本机机器码；
   无需写入磁盘或生成 DLL 文件。

3. **在内存中加载并执行**
   编译结果（函数指针）直接加载到内存；
   立即可调用，不需要 `LoadLibrary()` 或外部编译器。

4. **可动态释放或重编译**
   LLVM 提供内存管理，可以在运行时丢弃旧函数并生成新函数。

---

## ⚙️ SymEngine 使用的 LLVM 技术栈

| 模块                            | 功能         | 对应 LLVM API           |
| ----------------------------- | ---------- | --------------------- |
| **IRBuilder**                 | 生成 LLVM IR | `llvm::IRBuilder<>`   |
| **LLVMContext**               | 管理符号表      | `llvm::LLVMContext`   |
| **Module**                    | 存放函数定义     | `llvm::Module`        |
| **ExecutionEngine / ORC JIT** | JIT 编译执行   | `llvm::orc::LLJIT`    |
| **TargetMachine**             | 生成本机代码     | `llvm::TargetMachine` |

---

### ✅ 举个小例子：动态生成并执行 C++ 函数（无外部编译器）

```cpp
#include <llvm/ExecutionEngine/Orc/LLJIT.h>
#include <llvm/IR/IRBuilder.h>
#include <llvm/IR/LLVMContext.h>
#include <llvm/Support/TargetSelect.h>
#include <iostream>

int main() {
    llvm::InitializeNativeTarget();
    llvm::InitializeNativeTargetAsmPrinter();

    auto Context = std::make_unique<llvm::LLVMContext>();
    auto M = std::make_unique<llvm::Module>("jit", *Context);
    llvm::IRBuilder<> Builder(*Context);

    // 定义函数类型：int func(int, int)
    auto *FuncType = llvm::FunctionType::get(Builder.getInt32Ty(),
                                             {Builder.getInt32Ty(), Builder.getInt32Ty()}, false);
    auto *Func = llvm::Function::Create(FuncType, llvm::Function::ExternalLinkage, "add", M.get());

    // 函数体
    auto *BB = llvm::BasicBlock::Create(*Context, "entry", Func);
    Builder.SetInsertPoint(BB);
    auto args = Func->args().begin();
    llvm::Value *x = args++;
    llvm::Value *y = args++;
    auto *sum = Builder.CreateAdd(x, y);
    Builder.CreateRet(sum);

    // 创建 JIT
    auto JIT = cantFail(llvm::orc::LLJITBuilder().create());
    cantFail(JIT->addIRModule(llvm::orc::ThreadSafeModule(std::move(M), std::move(Context))));

    // 查找符号
    auto Sym = cantFail(JIT->lookup("add"));
    auto *addFunc = (int (*)(int, int))Sym.getAddress();

    std::cout << "add(3, 5) = " << addFunc(3, 5) << std::endl;
}
```

输出：

```
add(3, 5) = 8
```

✨ 这段代码**完全没有生成 DLL 文件**，所有代码都在内存中即时编译、即时执行。

## 🔍 LLVM JIT 与 Windhawk 式“动态编译”的关键区别

| 对比项  | LLVM JIT (SymEngine) | Windhawk 方式       |
| ---- | -------------------- | ----------------- |
| 编译目标 | LLVM IR → 内存代码       | C++ 源码 → DLL 文件   |
| 编译位置 | 内存中                  | 文件系统              |
| 加载方式 | 函数指针（无需 DLL）         | `LoadLibrary()`   |
| 依赖   | LLVM runtime         | 外部编译器（clang/MSVC） |
| 启动速度 | 极快（毫秒级）              | 快，但需调用外部进程（几百毫秒）  |
| 安全性  | 内存内沙箱（可限制 API）       | 需信任外部 DLL         |
| 灵活性  | 可重编译 / 生成代码          | 只能重新生成 DLL 文件     |

## 🔧 如果你想在 Launcher 中使用 LLVM 动态编译

你可以用 LLVM 实现一个“动态插件 JIT 系统”，思路如下：

### 1. 内置 LLVM Runtime

* 在你的项目中集成 LLVM（可通过 CMake FetchContent 或预编译包）；
* 初始化本机目标（`InitializeNativeTarget()`）。

### 2. 定义插件 API 接口（IR 层）

* 插件作者编写 C++ 类似脚本（或 DSL）；
* 你将这些代码解析为 LLVM IR（或由 Clang 前端生成 IR）。

### 3. 编译 IR 到机器码并执行

* 使用 `llvm::orc::LLJIT` 编译；
* 注册可调用函数符号（比如 `SearchFiles()`, `ShowToast()` 等）；
* 插件立即可运行。

### 4. 可热重载

* 插件代码更新 → 销毁旧模块 → 重新编译；
* 整个过程无需退出程序，也无需磁盘中间文件。
